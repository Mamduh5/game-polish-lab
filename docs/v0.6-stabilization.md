# Game Polish Lab v0.6 User Guide

Game Polish Lab is a VS Code visual polish console for Phaser-first game projects. It keeps polish work local to the workspace, records visual tuning choices under `.game-polish-lab/`, and uses guardrails so visual changes do not drift into gameplay, save, economy, progression, quest, hatch, merge, ad, level, or monetization logic.

The normal workflow is:

1. Choose a visual surface.
2. Preview and tune values.
3. Save config or asset choices.
4. Direct apply where a supported safe template exists.
5. Record the result as better, worse, same, mixed, or unreviewed.

Game Polish Lab is not an automatic game integration system. Unsupported or structural work should produce a scoped fallback task instead of broad source edits.

## Dashboard

Use `Game Polish Lab: Open Visual Tuning Dashboard` to inspect detected adapters, configs, recipes, field notes, scope status, direct-apply template status, rollback access, asset contract status, and optional dev overlay status.

Dashboard status is read-oriented. It should not install runtime files, broaden direct-apply scope, or make fallback work part of the normal tuning path.

## Preview Renderer

Use `Game Polish Lab: Tune Visual Surface` to compare before/after presentation for the selected surface. The preview renderer shows desktop/mobile frames, supported states, and supported animations where available. It is a visual preview, not a gameplay simulator.

## Style Presets

The style preset library provides named starting points for supported style surfaces. Presets are draft-only until saved. Saving creates or updates local config under `.game-polish-lab/styles/` and may direct apply only where the selected adapter has a safe template.

## Asset Contracts and Contact Sheets

Asset contracts live at `.game-polish-lab/assets/asset-contracts.json`. They describe expected asset slots, dimensions, transparency needs, and validation state.

Use `Game Polish Lab: Refresh Asset Contracts` to refresh contract data. Use `Game Polish Lab: Open Asset Contact Sheet` to inspect existing contract slots and preview current assets. Contact sheets are preview-only and should not patch loader or manifest code.

## Scope Guard

The scope guard classifies candidate paths before visual writes:

- Safe: `.game-polish-lab/**`, known visual style configs, generated visual modules, and known asset folders.
- Suspicious: scene, UI, renderer, loader, manifest, and broad source files that may be visual but need a guarded handoff.
- Forbidden: save, economy, reward, progression, unlock, upgrade, merge, hatch, quest, ad, monetization, analytics, level data, gameplay rules, and package manager files during visual writes.

Blocked or suspicious direct apply plans should stay non-executable. Use a scoped fallback task only when the work is unsupported, a one-time adapter setup, unusual integration, or structural work.

## Rollback History

Rollback snapshots live under `.game-polish-lab/rollback/`. Use `Game Polish Lab: Open Rollback History` to inspect snapshots and restore only scope-guard-safe visual files. Suspicious source or bridge files should create a fallback task instead of automatic restore.

## Direct Apply Templates

Direct apply templates are intentionally narrow. They write known safe style config paths and create rollback snapshots before overwriting existing files.

Asset replacement does not have an executable direct apply template. Asset copy or loader/manifest wiring must stay guarded and may require manual integration or a fallback task.

## Optional Dev Overlay

Use `Game Polish Lab: Create Optional Dev Overlay Spike` only when you want experimental developer tooling. The command asks for explicit approval before writing anything.

Generated files live under `.game-polish-lab/dev-overlay/`:

- `polish-dev-overlay.js`
- `polish-dev-overlay.css`
- `README.md`
- `manifest.json`

The overlay is inert unless the game URL contains `?polish=1` or `&polish=1`. It is manually integrated, developer-only, does not call network APIs, does not persist to game save data, and should be excluded from production builds unless intentionally included.

## Fallback Tasks

Fallback tasks are not the normal polish loop. Use them only for unsupported games, one-time adapter setup, unusual integration, or structural work that cannot be handled by safe config/direct-apply paths.

Fallback tasks must keep exact selected file scope, list suspicious and forbidden paths, and preserve the rule that gameplay, save, economy, progression, quest, hatch, merge, ad, level, and monetization files are not visual-polish targets.

## Known Limitations

- No executable direct apply template exists for `asset_replacement`.
- The dev overlay is experimental, manually integrated, and gated by `?polish=1`.
- Manual VS Code webview/dashboard testing may still be needed after code-level validation.
- Generic Phaser support is safe-config-first, not full automatic game integration.
- Direct applies are limited to known safe style config paths.
- Structural gameplay/layout changes are intentionally out of scope.
- Preview rendering is representative and does not replace in-game checks.
- Contact sheets preview asset contracts; they do not patch unknown loaders or manifests.
