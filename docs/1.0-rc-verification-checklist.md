# 1.0 RC Verification Checklist

Use this checklist for v0.99 RC verification. Manual VS Code smoke testing remains pending unless someone actually performs it in an Extension Development Host.

## Automated Checks

- `npm run compile` passes.
- `npm test` passes.
- `npm run lint` passes when present.
- `npm run package:check` passes.
- `npm run package` completes its dry-run package flow and does not publish.
- `git diff --check` has no whitespace errors.

## Product Workflow Checks

- Dashboard opens through `Game Polish Lab: Open Visual Tuning Dashboard`.
- Monster Farm adapter detection works.
- Sort Puzzle adapter detection works.
- Cursor Arena adapter detection works.
- Generic Phaser fallback path is visible when Generic Phaser is the practical adapter path.
- Surface rows generate for detected adapters.
- Config paths and existing config states are visible.
- The visual tuner labels preview source as real existing config, generated real-project default, fixture test preview, or disconnected example preview.
- The visual tuner shows `Save Config` for import-only/config-only/comment-only/disconnected states and reserves `Save & Apply` for connected `runtime_value_usage` proof.
- Direct apply status is accurate and does not claim unsupported runtime integration or import/comment/config-only connection.
- Dashboard and tuner show connection status, proof level, style source, evidence files, matched style properties, missing pieces, and whether runtime value usage enables direct apply.
- Setup Direct Apply patches recognized safe visual usage or declines and generates fallback guidance; unused imports are not accepted as setup success.
- Fallback task generation is clearly separated from direct apply.
- Scope guard warns or blocks suspicious and forbidden paths.
- Rollback discovery and restore paths stay inside the opened workspace.
- Asset validation and contact-sheet paths are usable where asset contracts exist.
- Result tracking and field notes remain reachable.

## Documentation Checks

- README links to RC docs.
- `docs/release-readiness.md` describes v0.99 as RC readiness and not publishing.
- `docs/1.0-release-candidate.md` lists frozen scope, deferred scope, pending manual decisions, and safety limits.
- `docs/release-notes-v1.0-rc.md` is factual and does not claim Marketplace release.
- Dogfooding docs still mention Idle Monster Farm, Sort Puzzle, and Cursor Arena.

## Known Limitations

- Publisher, license, icon, Marketplace copy, and manual VS Code smoke test are pending.
- Publishing to Marketplace is not part of v0.99 RC verification.
- Asset replacement has no generic executable direct-apply template.
- Sort Puzzle and Cursor Arena source/runtime integration remains fallback-only unless a project already reads generated config paths.
- Generic Phaser remains safe-config-first rather than full automatic integration.
- Generated webview previews are not live screenshots and are not direct-apply acceptance evidence.
- Fixture data is regression coverage only; real Idle Monster Farm acceptance must use the real project structure.
- Screenshot annotation and live dev overlay are not production v1.0 workflows.
- No new adapters, new visual surfaces, or gameplay/system editors are part of RC.
- No gameplay, save, economy, progression, ad, rules, solver, level-data, player/projectile, shooter, or monetization logic is touched during RC verification.
